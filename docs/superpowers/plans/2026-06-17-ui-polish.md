# Plan: UI 美化与交互改进

## 背景

Trawler Web UI 在 2026-06-16 完成重设计（Apple soft style），现在进入**打磨阶段**。用户反馈 3 个具体问题：

1. `/check` 页面平台卡片样式粗糙（E4 把进度条改成独立文字卡片后未细化视觉）
2. `/dashboard` 统计卡片的 tooltip 鼠标移入即消失，无法滚动查看长消息列表
3. `/dashboard` 数据未与 MessageStore 的 24h 窗口对齐

附带要求：全局设计一致性审查（圆角、阴影、暗色模式）。

## 设计原则（Apple soft style 要点）

本仓库已确立的设计语言，本次改动**严格遵守**：

| 维度 | 规则 | Token |
|------|------|-------|
| 表面 | 半透明 + `backdrop-blur-[12px]`，营造毛玻璃质感 | `--card-bg`, `--sidebar-bg` |
| 圆角 | 卡片 14px / 按钮与 badge 10px / 输入框 8px | `--radius-card`, `--radius-button`, `--radius-field` |
| 阴影 | 极柔（rgba 0.05 透明度），多档分明 | 暂未 token，本次补 |
| 字体 | `-apple-system` / SF Pro 栈，已在 `--font-sans` | tokens.css L34 |
| 字号层级 | 标题 `text-2xl font-semibold tracking-tight` / 数值 `text-3xl` / 正文 `text-sm` / 辅助 `text-xs` | — |
| 留白 | 卡片 `p-5`，section 间距 `mb-4`~`mb-6` | — |
| 状态色 | primary `#0071e3` / success `#34c759` / warning `#ff9500` / error `#ff3b30` | `--color-*` |
| 暗色模式 | 全部走 CSS 变量；硬编码 `bg-white`/`text-gray-*` 必须带 `dark:` 变体 | `@media (prefers-color-scheme: dark)` |

**反模式（本次要消除的）**：
- 硬编码 `bg-white dark:bg-gray-800` 而不走 `--card-bg`
- emoji 充当状态指示器（与 SVG 风格冲突）
- 圆角随意用 `[9px]`/`[6px]`/`[16px]` 破坏 token 体系
- tooltip 与触发器之间留 gap 导致 hover 断裂

  ```javascript
  function setPlatformStatus(platform, state, text) {
    var card = document.getElementById('card-' + platform);
    var bar = document.getElementById('bar-' + platform);
    var statusEl = document.getElementById('status-' + platform);
    var progEl = document.getElementById('progress-' + platform);
    if (!card || !bar || !statusEl || !progEl) return;

    // dot 色 + bar 色 + pulse + label + label 色 + 进度色
    var dotCls, barCls, pulse, label, labelCls, progCls;
    if (state === 'running') {
      dotCls = 'bg-orange-500'; barCls = 'bg-orange-500'; pulse = 'animate-pulse';
      label = '运行中'; labelCls = 'text-orange-500 font-medium';
      progCls = 'text-[var(--text-secondary)] font-mono tabular-nums';
    } else if (state === 'done') {
      dotCls = 'bg-green-500'; barCls = 'bg-green-500'; pulse = '';
      label = '完成'; labelCls = 'text-green-600 dark:text-green-400 font-medium';
      progCls = 'text-[var(--text-tertiary)] font-mono tabular-nums';
    } else {
      dotCls = 'bg-[var(--text-tertiary)]'; barCls = 'bg-[var(--text-tertiary)]'; pulse = '';
      label = '空闲'; labelCls = 'text-[var(--text-secondary)]';
      progCls = 'text-[var(--text-tertiary)] font-mono tabular-nums';
      text = '—';
    }
    bar.className = 'absolute left-0 top-0 bottom-0 w-1 ' + barCls + ' ' + pulse;
    statusEl.innerHTML =
      '<span class="w-2 h-2 rounded-full ' + dotCls + ' ' + pulse + '"></span>' +
      '<span class="' + labelCls + '">' + label + '</span>';
    if (text !== undefined) {
      progEl.textContent = text;
      progEl.className = 'text-[13px] leading-relaxed ' + progCls;
    }
  }
  ```

### Task 2: dashboard tooltip hover 断裂修复（P2）— 纯 CSS 桥接

- **文件**: `web/templates/dashboard.html`
- **当前问题**:
  - L8：`stat_tooltip` macro 的根 div 是 `absolute left-0 top-full mt-2 hidden group-hover:block`。`mt-2`（margin-top 8px）在卡片底边与 tooltip 顶边之间留出 8px 空白，鼠标从卡片移向 tooltip 时必然穿过这 8px，期间不在 `.group` 的命中区域内（`group-hover` 失活），tooltip 立即 `hidden`，无法 hover、无法滚动。
  - L9：tooltip 内 `max-h-72 overflow-y-auto` 本意支持滚动，但因进不去而失效。
- **设计方案（纯 CSS，零 JS）**:
  1. **移除 `mt-2`**，改用 `top-full`（tooltip 顶边紧贴卡片底边，无 gap）。
  2. **加透明桥接区**：在 tooltip 根 div 上加 `before:` 伪元素或用 `pt-2` padding（padding 是 hover 命中区，不会显示空白）。推荐 `pt-2` 方案——把 `mt-2` 的视觉间距挪到 tooltip 容器内部的 padding-top，padding 区域仍属于 tooltip 元素本身，鼠标穿过时仍在 group hover 内。
  3. tooltip 内层卡片（L9 的 div）保持原视觉。
  4. 确认 `z-50` 不被相邻卡片遮挡（相邻 `.relative.group` 无 `z-index`，tooltip 的 `z-50` 足够）。
- **改动**:

  **before**（L8-9）:
  ```html
  <div class="absolute left-0 top-full mt-2 hidden group-hover:block z-50 w-72">
    <div class="bg-white dark:bg-gray-800 border border-[var(--card-border)] rounded-[10px] shadow-[0_8px_24px_rgba(0,0,0,0.12)] p-2 max-h-72 overflow-y-auto">
  ```

  **after**:
  ```html
  <div class="absolute left-0 top-full hidden group-hover:block group-focus-within:block z-50 w-72 pt-2">
    <div class="bg-[var(--card-bg)] backdrop-blur-[12px] border border-[var(--card-border)] rounded-[10px] shadow-tooltip p-2 max-h-72 overflow-y-auto">
  ```

  **说明**:
  - `mt-2` → `pt-2`：桥接区从 margin（外部，不属 tooltip）变 padding（内部，属 tooltip），鼠标穿过时仍在 hover 命中区。
  - 加 `group-focus-within:block`：键盘可访问性（Tab 到卡片也能展开 tooltip）。
  - `bg-white dark:bg-gray-800` → `bg-[var(--card-bg)] backdrop-blur-[12px]`：顺带修 P6 暗色模式硬编码，与全站卡片同源。
  - `shadow-[0_8px_24px_rgba(0,0,0,0.12)]` → `shadow-tooltip`：用 Task 5 新增的 token。
- **额外验证点**:
  - 卡片与 tooltip 之间**视觉上仍有 8px 间距**（由 `pt-2` 提供），但鼠标可以平滑穿越。
  - tooltip 内 `overflow-y-auto` 在消息 >10 条时出现滚动条，鼠标在 tooltip 内滚动不触发外层页面滚动。
  - 鼠标从 tooltip 移出（向左/右离开）时 tooltip 正常消失（离开 group）。
  - 鼠标在 tooltip 内停留时 tooltip 不闪烁。
- **验证**:
  - 制造 >10 条消息（临时往 messages.json 塞数据或跑一次 check）
  - 鼠标从"错误"卡片移到 tooltip，tooltip 不消失
  - 在 tooltip 内滚轮滚动，能看完全部消息
  - 鼠标移开 tooltip（向下/左/右），tooltip 消失
  - 键盘 Tab 到卡片，tooltip 展开（`group-focus-within` 生效）

### Task 3: dashboard 统计对齐 24h 窗口（P3）

- **文件**: `web/routes/dashboard.py`, `shared/message_store.py`
- **当前问题**:
  - `dashboard.py` L23：`all_msgs = store.get_messages()` 取**全量**消息，无窗口过滤。
  - `message_store.py` L231 `cleanup()` 会删除超 24h 消息，但**只在 pipeline 主动调用时**才清；dashboard 路由既不调 cleanup 也不做窗口过滤，导致统计可能包含远超 24h 的陈旧数据（如果用户长时间没跑 check，cleanup 没触发）。
  - dashboard 副标题 L38 写"实时概览"，但实际是"全量历史概览"，语义不符。
- **设计方案**:
  1. 在 `MessageStore` 加 `get_messages_in_window(window_hours=24, **filters)` 便捷方法，复用 `is_in_window`，避免在路由里重复过滤逻辑。
  2. dashboard 路由改用新方法取 24h 内消息，所有统计（total/active/pushed/error/recent/tooltip sample）基于窗口内数据。
  3. **不在 dashboard 路由里调 `cleanup()`**——cleanup 是写操作（改磁盘），Dashboard 是只读视图，混入写操作会导致每次刷新都触发 IO 且可能与其他正在跑的 pipeline 竞争。窗口过滤是只读的、安全的。
- **改动**:

  **`shared/message_store.py`** — 在 `get_messages` 方法后（L137 后）新增:
  ```python
  def get_messages_in_window(
      self,
      window_hours: int = DEFAULT_WINDOW_HOURS,
      *,
      phase: Phase | None = None,
      exclude: bool = False,
      platform: str | None = None,
  ) -> list[MessageRecord]:
      """获取时间窗口内的消息（默认 24h），支持按阶段和平台过滤。

      与 ``get_messages`` 的区别：只返回 ``pubdate`` 在窗口内的消息，
      不会删除超期数据（只读，安全用于 dashboard 等只读视图）。
      实际清理由 ``cleanup()`` 在 pipeline 中按需调用。
      """
      cutoff = time.time() - window_hours * 3600
      results: list[MessageRecord] = []
      for msg_id, data in self._messages.items():
          if data.get("pubdate", 0) < cutoff:
              continue
          if platform is not None and data.get("platform") != platform:
              continue
          msg_phase = data.get("phase", "")
          if phase is not None:
              if exclude and msg_phase == phase.value:
                  continue
              if not exclude and msg_phase != phase.value:
                  continue
          results.append(self._msg_from_dict(msg_id, data))
      return results
  ```

  **`web/routes/dashboard.py`** L23 改动:
  ```python
  # before
  all_msgs = store.get_messages()

  # after
  all_msgs = store.get_messages_in_window()  # 默认 24h，只读安全
  ```

  其余 L26-61 统计逻辑**不动**（都基于 `all_msgs`，自动随窗口收窄）。
- **验证**:
  - 手动往 `data/messages.json` 塞一条 `pubdate` 为 48h 前的消息，刷新 dashboard，该消息不出现在统计数和 recent 表
  - 跑一次 check 触发 pipeline 的 cleanup，确认超期消息被删除（不影响 dashboard 行为，dashboard 本就只读过滤）
  - dashboard 统计数 ≤ 实际 messages.json 总条数（窗口收窄后必然）
  - 单测：新增 `tests/test_message_store.py::test_get_messages_in_window`，构造 3 条消息（1h 前 / 25h 前 / 48h 前），断言默认窗口只返回 1 条，窗口=48 返回 2 条
